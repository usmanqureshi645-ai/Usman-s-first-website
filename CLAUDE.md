# Project: Usman's Personal Website

A single-page personal/professional site for Usman (ACCA, ACA — Big 4 audit & advisory) with several AI-powered interactive tools built on top of a static HTML/CSS/JS frontend and Vercel serverless functions.

## Stack
- Frontend: three distinct pages — `home.html` (marketing landing page, `/` redirects here via `vercel.json`), `index.html` (the "body": all AI tools + the four tabs AI Tools → Hot Market Topics → My Workspace → Who am I), and `workspace.html` (logged-in workspace). `index.html` is mirrored to `Latest Business card.html` — **always `cp index.html "Latest Business card.html"` before committing**, they must stay byte-identical. **⚠️ NEVER copy `home.html` onto `index.html` (or vice-versa).** They are different files. Doing so wipes out the tools body — this exact mistake ("Copy home.html to index.html") clobbered the body three times on 2026-07-04 and had to be restored from git commit `4a03876`. The ONLY mirror relationship is `index.html` ↔ `Latest Business card.html`.
- Backend: `/api/*.js` serverless functions on Vercel, calling the Anthropic API (model `claude-sonnet-4-6`)
- Hosting: Vercel project `usman-s-first-website`, auto-deploys on push to `main` on GitHub (`usmanqureshi645-ai/Usman-s-first-website`)
- Email: Resend API
- Persistence: Upstash Redis (REST API) — feedback log, Meeting Room/Knowledge Test conversation state for email reply-to-continue (30-day TTL, keys `mrconv:<uuid>` / `qzconv:<uuid>`), permanent user accounts (`user:<email>`), permanent per-user consultation history (`consultations:<email>` list, capped at 200) — see Accounts & Workspace below — and anonymous per-IP usage tracking (`iplog:<ip>`, no TTL, see Anonymous usage tracking below). Two more lists feed the weekly Excel export (see `lib/featureLog.js`/`lib/exportData.js` below): `feature_usage_log` (RPUSH, cap 5000 — one entry per genuine, non-test completion of Meeting Room/Knowledge Test/GAAP Compare/CV Scoring/CV Reasoning/FS Review/Resume Session) and `feature_ratings_log` (RPUSH, cap 5000 — one entry per submitted feature rating). `signups_log`'s cap was raised from 500 to 4999, since it's now Sheet 1's source of truth, not just a dashboard preview.
- Accounts & Workspace: real email+password accounts (zero external auth dependencies — `lib/auth.js` hand-rolls password hashing via Node's `crypto.scrypt` and a small HMAC-signed session cookie, no bcrypt/jsonwebtoken). **Hard-gated (changed 2026-06-29): every AI tool now requires signup/login before use, enforced both client-side and server-side.** Anonymous visitors can still see each tool's panel and do free preparatory actions (pick Meeting Room personas, upload/extract a CV or financial statement's text client-side, pick a GAAP jurisdiction or FS Review framework, type into an input box) — none of that reaches an API — but the actual AI-invoking action (`mrStartMeeting()`, `qzStart()`, `gaapSendMessage()`, `jobReviewCv()`, `fsSubmitReview()`, `window.sendAsk()`) opens the signup widget instead of proceeding if `!window.getSiteUser?.()`, via the same `if(!window.getSiteUser?.()){ window.toggleSignup?.(true); return; }` pattern originally used only by `gaapSaveConsultation()`. Every one of the 7 AI endpoints (`meeting.js`, `quiz.js`, `gaap.js`, `ask.js`, `tailor.js`, `cv-review.js` for `score`/`chat` modes, `fsreview.js`) independently enforces this server-side via `getUserFromRequest(req)` → 401 if absent, so the gate can't be bypassed by calling the API directly. `cv-review.js`'s `mode:'validate'` (the cheap upload-time document-type check) is deliberately exempt, since it's a preparatory action, not a scoring action. `fsreview.js` no longer accepts a client-supplied `email` body field — it always emails the logged-in user's own session address now. The former "soft-gate" nudge system (`window.showSignupNudge()`, anonymous-email `prompt()` fallbacks in `mrEndMeeting()`/`qzEndSession()`/`gaapEndSession()`) was removed as dead code, since nothing can reach those post-result paths without already being logged in. Logged-in users get consultations auto-saved (Meeting Room, Knowledge Test, Financial Statement Review, CV Scoring) or manually saved via a "💾 Save to my workspace" button (GAAP Compare; AskAI not yet wired). `workspace.html` is a separate static page listing saved consultations with a "Continue this conversation" action that round-trips through `index.html?resume=<id>&tool=<tool>` — it already hard-gated its own content behind a login check at page load before this change, so it needed no changes here. **Signup now collects 5 additional mandatory fields** beyond name/email/password: `company` (text), `department`/`departmentOther` and `designation`/`designationOther` (each a searchable dropdown + its own dedicated "Other" free-text box — never shared between the two), `country` (searchable dropdown, embedded ~195-entry static list in `index.html`), `city` (plain text — no feasible exhaustive world-city dataset). All stored directly on `user:<email>` (not via `lib/profile.js`, a separate optional post-signup system) and mirrored into `signups_log`. A duplicate-email signup now returns `{error:'This account has already been used', duplicate:true}`; the frontend's `submitSignup()` checks `data.duplicate` and shows an inline recovery link (reusing the existing forgot-password flow) instead of an alert.
- Anonymous usage tracking: every AI endpoint still calls `logAndCheckUsage()` from `/lib/ipUsage.js` (after the login gate above, so it only ever sees already-authenticated requests now), which logs the requesting IP (`x-forwarded-for`) to Upstash under `iplog:<ip>` (count + tools used, no TTL) and returns `{ count, shouldNudge }`. The rate-limiting half of this (20 req/60s per IP) is still load-bearing anti-abuse. `shouldNudge` and the per-IP unique/repeat split are now vestigial — nothing reads `shouldNudge` since `window.showSignupNudge()` was removed, and "unique/repeat per IP" now effectively means "per logged-in user" since every caller is authenticated; left as-is rather than refactored (see `lib/metrics.js`/`dashboard.html` note below).
- Genuine feature-usage logging: `/lib/featureLog.js` is a SEPARATE system from `lib/metrics.js`'s `recordToolUse`/`TOOLS` above — that one counts every single AI message regardless of whether it was a real session; this one only logs a "completion" when an action was both explicitly finished (End-session button / Score button / Review button clicked) AND judged genuine rather than a quick test, via `isMeaningfulSession(transcript, {tool})` (a simple per-tool threshold on user-message count + combined character length). Called from `send-summary.js`/`qz-summary.js` (Meeting Room, GAAP, Knowledge Test — gated by `isMeaningfulSession`, plus a parallel `tool:'resume-session'` entry when a `resumed:true` flag is passed), `cv-review.js` (score mode logs unconditionally on success since a real-document gate already ran; chat mode logs once, exactly when the user's message count hits 2 — see the risk note on that file below), and `fsreview.js` (logs unconditionally on a successful review, since reaching that point already means a real financial-statement document was processed). `lib/exportData.js`'s `buildWorkbook(kv)` reads this list (plus `signups_log` and `feature_ratings_log`) and builds the 3-sheet Power-BI-ready `.xlsx` (Signups / Feature Usage / Feature Ratings) — always a full snapshot, never incremental, and **deliberately excludes password/passwordHash from the Signups sheet** regardless of the original requirements doc listing "Password" as a signup field. Emailed weekly via a new `vercel.json` cron (`?action=cron-export-data`, Mondays 10:30 UTC, distinct from the `cron-followups` daily job), and available on demand via `?action=export-data` (GET, `DASHBOARD_KEY`-gated like `?action=dashboard`, returns the `.xlsx` directly as a file download).
- Feature-rating popup: a new floating card (mirrors the USP-toast IIFE's construction, busy-state check, and "Not now" dismiss semantics) appears 20 minutes after page load and is re-eligible at most once per hour — tracked via a `localStorage` timestamp rather than `sessionStorage`, since the hourly cooldown must survive reloads/tab changes (the USP toast's own session-only cap stays as-is). Asks which of the 6 tracked AI tools the visitor is enjoying most + a 1-5 rating, with a "Report a bug" link into the existing feedback widget. Submits to `feedback.js`'s new `?action=rate`.
- Server-side npm dependencies: `package.json` (`"type": "module"`) was added for the first time to bring in `jszip`, used by `/lib/docxComments.js` to inject real OOXML Word comments into uploaded `.docx` files (or a freshly built one, for PDF/txt uploads) — see Financial Statement Review below. `exceljs` was added the same way for `/lib/exportData.js`'s `.xlsx` workbook building (pure-JS, no native bindings, same no-build-step compatibility as jszip). Everything else server-side remains dependency-free; file-text-extraction (pdf.js/mammoth) stays client-side via CDN as before.
- Email sending domain: `uqconsulting.org`, verified in Resend (SPF/DKIM/DMARC + inbound MX to `inbound-smtp.eu-west-1.amazonaws.com`) — replaced the old `onboarding@resend.dev` sandbox sender, which could only deliver to the account owner's own address, not visitor-typed emails
- Owner's contact email: `usmanqureshi645@gmail.com` (used as the destination for signup/feedback notification emails)

## Env vars (set in Vercel, production)
- `ANTHROPIC_API_KEY` — powers every `/api/*.js` AI endpoint
- `RESEND_API_KEY` — outbound email (signup welcome, feedback log, meeting/interview summaries)
- `UPSTASH_REDIS_REST_URL`, `UPSTASH_REDIS_REST_TOKEN` — feedback persistence + Meeting Room conversation state (`/api/feedback.js?action=list` reuses the token itself as a simple shared-secret query param for read access)
- `RESEND_WEBHOOK_SECRET` — optional, not yet set; if added, `/api/inbound-email.js` will verify Resend's Svix webhook signature instead of trusting payloads by UUID-unguessability alone
- `AUTH_SECRET` — **must be set in Vercel** for accounts to work at all (`lib/auth.js` throws if missing); any long random string, used to HMAC-sign session cookies. Not yet generated/set as of this writing — do this before relying on signup/login in production.
- `CRON_SECRET` — optional; if set, `/api/cron-followups.js` requires `Authorization: Bearer <secret>` (Vercel doesn't inject this automatically — only useful if you add it as a custom header via Vercel's cron config or call it yourself)

## API endpoints (all in `/api/`)
- `meeting.js` — Meeting Room panel chat. Personas selectable by the user (Sarah Bennett/audit, David Whitfield/IFRS-FRS, Amelia Hartley/tax, Marcus Lee/forensic, Elena Rossi/ESG, James Carter/legal, Charlotte Sinclair/technical accounting). Greets → asks name → interactive discussion with cross-references, risk/mitigation exchanges, clarifying questions, redirects if off-track. Shared persona/system-prompt logic lives in `/lib/meetingPersonas.js` (also used by `inbound-email.js`). Users can also add their own custom specialist(s) ("+ Add your own specialist" in the picker panel) with a free-text name + role and a male/female avatar (generic stock photos `images/team/generic-male.png` / `generic-female.png`); `buildMeetingSystemPrompt()` accepts `agents` as a mix of built-in name strings and `{name, role}` objects for these. Frontend warns (but doesn't block) if the typed name matches a built-in persona already in the roster, suggesting the user just tick that box instead. Custom personas can be added/removed mid-conversation like built-ins, and round-trip through save/resume (`record.agents`), though resumed custom personas default back to a female avatar (gender isn't persisted). **Experimental — live two-person sessions**: a logged-in host can click "+ Invite a real person" to bring a real colleague into the same Meeting Room session live. No websocket infra exists on this stack, so this is polling-based (`lib/meetingSession.js` stores the shared session in Redis under `mrsession:<id>`, 6h TTL; both browsers poll `account.js?action=meeting-session` every ~5s). Routed through `account.js` (not a new `/api/*.js` file, to respect the 12-function cap): `?action=meeting-invite` (creates/reuses a session, seeds it with any conversation so far, emails the invitee a join link — straight in if they already have an account, else a signup link first), `?action=meeting-join` (marks the invitee present + returns full history, called on page load when `?meetingjoin=<id>` is in the URL), `?action=meeting-post` (append a message), `?action=meeting-end` (marks ended). Whoever sends a human message also calls `/api/meeting` themselves and posts the AI reply back to the shared session — there's no server-side orchestrator, so there's a small race window if both people type at the same moment. `send-summary.js` accepts an optional `coParticipant:{email,name}` to email both people the joint summary and save it to both workspaces, noting they attended together. **This was explicitly framed as a "let's see if it's fun" experiment — if it doesn't hold up in practice, it's fine to rip out** (frontend: the "+ Invite a real person" block + `mr*Session*`/`mr*Poll*` functions in `index.html`; backend: `lib/meetingSession.js` + the `meeting-*` actions in `account.js` + the `coParticipant` bits in `send-summary.js`).
- `inbound-email.js` — Resend inbound webhook receiver. Lets users reply to their Meeting Room summary email to resume the conversation: looks up the conversation by the `meetingroom+<uuid>@uqconsulting.org` address the reply was sent to, replays it through the same panel persona prompt, emails the panel's reply back, and re-persists the updated history (30-day TTL).
- `gaap.js` — GAAP Compare research bot, jurisdiction-aware (US/UK/Ireland/Luxembourg/Australia).
- `quiz.js` — Knowledge Test interview coach. Onboarding flow: greet as a friend → ask role/JD → ask experience → mention CV tool exists separately → mention email feedback → then ask questions. Tone must always stay kind/encouraging, never harsh.
- `tailor.js` — **cover-letter writer only** (CV generation was removed). Writes one role-specific cover letter from the candidate's CV + job posting, with an optional `companyUrl` the function fetches server-side (8s timeout, HTML stripped to ~6000 chars, silent fallback to model knowledge) so the motivation reflects something real about the firm. Strong anti-"AI-sounding" language rules baked in (plain UK business English, no em/en dashes, no generic AI filler, no invented achievements). Returns `{ coverLetter }`. The old `===CV===` generation path is gone.
- `cv-review.js` — CV/cover-letter **scoring** (no document generation). `mode:'score'` (default) returns structured JSON: `{ cv:{ hr, ats, hiringManager, items:[{criticism,suggestion}] }, coverLetter: null | {same shape} }` — three reviewer perspectives (HR screener / ATS / hiring manager) and a criticism-paired-with-fix list, rendered as two-column scorecards on `#jobhub` and in the workspace CV tab. `mode:'chat'` powers the workspace "Talk through it with Eleanor" coach — a named persona **Eleanor Hughes** (British recruitment specialist, avatar `images/team/cv-reviewer.png`), highly interactive: invites the user's pushback, adapts suggestions to their constraints (e.g. if revenue is confidential, suggest a range, then other impact measures), **never repeats a rejected suggestion**, respects their final position. Takes `{ messages, cv?, coverLetter?, jobDescription? }`, returns `{ reply }`, plain human language, no markdown, coaches rather than rewrites. The workspace chat has Speak (STT) / Hear (TTS) / Attach, powered by a **self-contained `VoiceEngine` + `wireMic` copied into `workspace.html`** (the originals live in `index.html`; workspace.html had no voice before). Deliberately the opposite tone to quiz.js. **No more CV/Word/PDF generation anywhere** — the `#jobhub` "Score My Resume" section is scoring-only, and a CTA ("I've made my edits, review them again") routes logged-in users to `workspace.html?tab=cv&review=1` to continue in the review chat. **Scoring track record:** every scoring (from `#jobhub` or the workspace) auto-saves for logged-in users via `account.js?action=save-consultation` (`tool:'cv-review'`, the rendered scorecard stored as `summaryHtml`); the workspace CV tab lists them as one-line dated links that expand the saved scorecard (and they also show in the History tab). **`mode:'validate'`** is a cheap upload gate (`{ text, kind:'cv'|'cover' }` → `{ isValid, detectedType }`) called by `jobHandleFile`/`cvHandleFile` to refuse the wrong document (e.g. financial statements uploaded as a CV) before scoring; it fails open on errors. The reviewer chat persona is **Eleanor Hughes** (British; avatar `images/team/cv-reviewer.png`, a copy of priya-nair.png decoupled from the name). The chat is **strictly scoped** to CV/cover-letter improvement: uploading a non-CV (financial statements etc.) makes her politely decline and ask how the user wants to use that info on their CV, then help word it; a real CV/cover gets criticism + suggestions. **Continuity:** scorings save the JD + a plain-text `priorScoring` (in the consultation's `transcript` object); the chat seeds from the latest scoring via a `priorScoring` chat-body field and pre-fills the JD, so it continues from stage one. Chat uploads (`cvChatHandleFile`) do NOT dump extracted text into the box (show a chip, read in background, auto-send). Eleanor replies in clean business English with only sparing `**bold**`, rendered client-side (`cvFormatReply`: `**x**`→`<strong>`, stray markdown stripped). The "Write me a cover letter" block has its own JD box (`cvCoverJobDesc`). **Workflow + continuity (2026-06-28):** the CV tab (renamed "My CV & Cover Letter Scoring", matching `#jobhub`'s renamed "Score My Resume") shows a visible workflow explainer (upload → score → challenge with Eleanor → re-upload → score again) and is laid out as two columns (`.cv-grid`: 380px upload+history rail, then the score/cover-letter/chat workspace) inside a widened workspace page (`main` max-width `min(1760px, 96vw)`). Every evaluation has a "Challenge this evaluation with Eleanor" action — after a fresh scoring, and on each numbered entry ("Evaluation #1", "#2"...) in the scoring history — via `cvGoToEleanor(priorScoringText, jdText, label)`, which sets `cvLatestPriorScoring`, pre-fills the JD box, scrolls to the chat and drops a bridging message, so an evaluation and the discussion about it are never disconnected. The workspace chat has a **focus/full-screen mode** (`cvToggleMaximize` reparents `#cvReviewChat` to `<body>`, hides the rest, and pauses the workspace music via `window.wsMusicSuppress`/`wsMusicRelease`), and shows a "voice may sound robotic, real accents coming soon" note when Hear is on. **Gotcha:** the score branch's `rawText` must NOT be named `text` — a `let text` there would shadow the destructured `text` body param across the whole try block and put the validate branch in a temporal dead zone. **Genuine-use logging risk:** the chat mode's `cv-review-chat` completion log fires exactly when the incoming `messages` array's user-message count hits 2 — this relies entirely on `workspace.html`'s `cvChatMessages` always resending the full, never-truncated history (confirmed true today; chat isn't resumable, so a page reload starts a fresh array and could in principle re-trigger the count-of-2 moment for what's arguably a new session anyway). If `cvChatMessages` is ever capped/truncated client-side in the future (e.g. to control token costs), this metric could mis-fire — re-check `lib/featureLog.js`'s `isMeaningfulSession` usage notes if that change is ever made.
- `ask.js` — general-purpose "AskAI"/"[Name]'sGPT" assistant, not finance-specific.
- `fsreview.js` — Financial Statement Review. Upload financial statements (.pdf/.docx/.txt, extracted client-side as usual), pick a framework (FRS 101/FRS 102/Full IFRS/US GAAP/AICPA), and the Meeting Room panel (`buildMeetingSystemPrompt()` from `/lib/meetingPersonas.js`, overridden with a structured-JSON review task instead of its normal conversational format) reviews the text for compliance discrepancies, cross-reference mismatches, and wording/spelling issues. Discrepancies anchored to exact quoted text become real, native Word comments authored "Usman Qureshi" via `/lib/docxComments.js` (injected into the original `.docx` if one was uploaded, or a freshly built one for PDF/txt uploads — PDFs can't carry real Word comments, so those get rebuilt as a clean, commentable `.docx`). Broader risks/reservations that can't be anchored to specific text go in the email body as additional notes instead. The annotated document is always emailed (Resend, base64 attachment) regardless of signup status; logged-in users additionally get the review auto-saved to their workspace (`tool: 'fsreview'`).
- `send-summary.js` — emails a discussion summary for either Meeting Room or GAAP Compare's "End session" button (`kind:'meeting'|'gaap'`, both routed through this one file rather than a new `/api/*.js` — a small `KIND_CONFIG` lookup varies the email framing/from-address/subject and the "Panel"/"GAAP Champion" speaker label; `kind==='meeting'` additionally gets the `mrconv:<uuid>` reply-to-continue persistence and the auto-save-to-workspace block, both deliberately scoped to that kind only since GAAP already has its own separate `gaapSaveConsultation()` save path). After a successful send, logs a genuine-completion entry (see Genuine feature-usage logging above) gated by `isMeaningfulSession`.
- `qz-summary.js` — emails a kind, specific, never-harsh interview-prep feedback report (Knowledge Test's "End session" button). Same genuine-completion logging as `send-summary.js`.
- `feedback.js` — routed by `?action=`: default (POST) is technical-agent assessment of submitted feedback (valid/not), persisting every submission to Upstash Redis (`feedback_log` list, capped at 500, each entry has a unique `id`) and emailing a log entry to the owner regardless of validity. Captures the reporter's email automatically if they're logged in (`lib/auth.js` session), or offers an optional "notify me" prompt in the widget for anonymous reporters, saved via `?action=attach-email` (POST, `{id, email}`). `?action=list` (GET) lists stored feedback (`?key=<UPSTASH_REDIS_REST_TOKEN>&limit=N`) — merged from the former standalone `feedback-list.js` to free up a function slot for `fsreview.js` under the Hobby-plan cap. `?action=resolve` (POST, same `?key=<UPSTASH_REDIS_REST_TOKEN>` auth, body `{id, resolutionSummary}`) marks an entry resolved and, **only if it has an email on file**, sends the reporter a personal thank-you email quoting what they reported and what was done about it. **Workflow: after fixing a bug that came in through the feedback widget, always call this resolve endpoint** so the reporter gets closure; don't just fix silently. `?action=rate` (POST, `{rating, feature, email?}`, public/anonymous-friendly like the rest of this file) is the new feature-rating-popup submission, logged via `lib/featureLog.js`'s `logFeatureRating`.
- `account.js` — **consolidated** accounts + workspace endpoint, routed by `?action=`: `signup`, `login`, `logout`, `me` (session cookie auth), `save-consultation` (login-required; lets GAAP Compare and, when wired, CV tools/AskAI save a raw transcript to the permanent workspace history), `workspace-list` / `workspace-get` (login-required; power `workspace.html`'s history list and "Continue this conversation" resume flow), `personalized-jobs` (login-required; powers `workspace.html`'s "Job Search" tab — takes CV/cover-letter text plus location/country/experience-level filters, calls Claude to extract likely role titles + a short fit summary, then builds pre-filled LinkedIn/Indeed/TotalJobs/eFinancialCareers search links server-side; deliberately no scraping or third-party jobs API, same ToS-avoidance rationale as the old Job Search quick-links), `cron-followups` (called by the daily Vercel Cron in `vercel.json`, 09:00 UTC — pulls due entries from the `followups_zset` sorted set and emails a 24h "did this help?" check-in, with reply-to-continue for Meeting Room/Knowledge Test only; runs once daily so delivery can lag up to ~24h past the nominal mark), `cron-export-data` (weekly Vercel Cron, Mondays 10:30 UTC, same `Bearer CRON_SECRET` pattern as `cron-followups` — emails the 3-sheet Excel export to the owner), and `export-data` (GET, `DASHBOARD_KEY`-gated like `?action=dashboard` — same export, returned as an on-demand file download). **Why one file**: see the Hobby-plan function-count constraint below — this used to be 8 separate files (`signup.js`, `login.js`, `logout.js`, `me.js`, `save-consultation.js`, `workspace-list.js`, `workspace-get.js`, `cron-followups.js`) until that broke production deploys.

## Admin analytics dashboard (`dashboard.html`)
Owner-only, static page (not a function), guarded by the same Upstash-token shared-secret as `feedback.js` (`?key=<UPSTASH_REDIS_REST_TOKEN>`, prompted once and cached in `localStorage`). Shows live traffic, visitor counts, signups database, per-AI-feature usage (unique vs repeat), Resend/Vercel free-tier gauges, and client-side API health pings. Powered by:
- `lib/metrics.js` — all analytics helpers over Upstash REST `/pipeline` (recordVisit/recordPresence/recordEmail/recordToolUse/recordSignup/getDashboardData/claimEmailAlerts). Redis keys: `metrics:visits:*`, `metrics:uniq:*` (HLL), `presence_zset` (live = ZCARD within 2 min), `metrics:signups:total` + `signups_log`, `metrics:tool:<tool>:{total,users,repeat}`, `metrics:email:*`, `metrics:invocations:<month>`, `metrics:alert:email:{day,month}:*` (SET NX claim flags). **Tool keys mirror the strings passed to `logAndCheckUsage` — keep `TOOLS` in sync.** `getDashboardData` also returns a 7-day `trend` (per-day visits + unique) rendered as an inline SVG chart on the page.
- `lib/email.js` — `sendEmail(resendKey, payload, kv)` is now the single send path for **every** outbound Resend email (replaced all raw `fetch('https://api.resend.com/emails')` POSTs); it counts the send. New email code MUST use it, not raw fetch, or the email gauge undercounts. After each successful send it also runs `maybeAlertOwner` → `claimEmailAlerts` and emails the owner **once per day/month** when Resend usage crosses **80%** of the free tier (80/day, 2400/month); that alert is sent via raw fetch so it doesn't recurse or count itself. (metrics.js must NOT import email.js — keep the dependency one-way to avoid a cycle.)
- `account.js` `?action=track` (public; page-load + visibility-gated heartbeat from `index.html` feed live/visitor counts) and `?action=dashboard` (GET, key-protected; returns `getDashboardData`). Both already count toward the 12-function total via `account.js` — no new function file was added.
- Vercel/Resend numbers are **self-counted estimates** (no Vercel/Resend usage API is wired); the invocation count is an activity proxy, labelled as such on the page.


## Page structure (top to bottom)
Hero → USP tools showcase → **Meeting Room → Knowledge Test → Score My Resume (`#jobhub`, formerly "Looking for a Job", then "Rate Your Resume") → GAAP Compare → Financial Statement Review → Hot Market Topics** (the 6 interactive AI tools, intentionally placed right after the hero) → Trust → Services → About → Sectors → Standards → Skills → Client Experience → Publications → Trainings → Contact → Footer. The USP showcase grid has a 7th tile, "Personalized Job Search For You", which doesn't scroll to a section — it sends logged-in users straight to `workspace.html?tab=jobsearch` (via `window.openPersonalizedJobSearch()`) or prompts a free signup first.

## Floating widgets (bottom-right, stacked)
Audio toggle (ambient music), Welcome/Site Guide bot, Feedback widget, Sign-up/Login widget, AskAI/personal GPT. All five are mutually exclusive — opening any one closes the others via `window.closeOtherWidgets()` (also cancels any in-progress TTS). The Sign-up widget doubles as a login form (toggle link inside the panel switches modes) and, once logged in, its floating button becomes a shortcut to `workspace.html` instead of reopening the signup form. A separate, non-mutually-exclusive **feature-rating popup** (see Genuine feature-usage logging above) also gets closed by `closeOtherWidgets()` but isn't part of the five-widget toggle set itself.

## Known constraints / conventions
- **Vercel Hobby plan caps a deployment at 12 Serverless Functions, hard limit.** Every `.js` file directly inside `/api/` counts as one function — exceeding 12 makes the ENTIRE deployment fail silently at the "Deploying outputs" stage (the build itself succeeds, logs show no obvious error, and Vercel just keeps serving the previous production deployment, which is very confusing to debug). Currently sitting at exactly 12 (`account.js`, `meeting.js`, `inbound-email.js`, `gaap.js`, `quiz.js`, `tailor.js`, `cv-review.js`, `ask.js`, `fsreview.js`, `send-summary.js`, `qz-summary.js`, `feedback.js`). **Before adding any new `/api/*.js` file, either delete/merge an existing one or route the new action through `account.js`'s (or `feedback.js`'s) `?action=` pattern instead.** Shared logic belongs in `/lib/` (not routed, doesn't count). To diagnose a deploy that looks "stuck" on old code: `vercel inspect <deployment-id> --logs` (requires `vercel whoami` to already be authenticated; PATH may need `$env:Path += ";C:\Program Files\nodejs;C:\Users\khani\AppData\Roaming\npm"` in a fresh PowerShell session) or check `https://api.github.com/repos/usmanqureshi645-ai/Usman-s-first-website/commits/<sha>/status` for a quick pass/fail without needing the CLI at all.
- Don't scrape or reproduce third-party copyrighted content (e.g. Big 4 "at a glance" guides) — build original equivalents instead (e.g. the hand-built IFRS flowchart diagrams).
- Job-related tools deliberately do NOT scrape LinkedIn/Indeed or auto-submit applications (ToS risk). `#jobhub` ("Score My Resume") only does AI CV/ATS scoring + tailoring/upload; the workspace "Job Search" tab (`personalized-jobs` action) only generates AI-curated, pre-filled search links from the CV — never live scraped listings.
- New interactive AI tools follow the existing pattern: a `mr-chat`-style UI in `index.html` + a matching `/api/<name>.js` serverless function with its own system prompt.
- Voice features use the shared `VoiceEngine` (browser TTS, male/female selectable) — no paid voice API wired in yet; would need ElevenLabs if the user ever wants real human-cloned voices (they have recordings ready if so).
- Background music is a shuffled playlist of 3 real royalty-free piano/flute/ambient tracks in `/audio/`, fixed-position office-scene SVG background behind all content (cursor parallax).
- The "visits/signups" counters on the hero are illustrative (day-seeded random, signups always < visits) — not real analytics. Signing up does bump the displayed count for that session.
- `git add -A` will break on the `Previous feedbacks/` folder (long `.eml` filenames exceed Windows path limits) — it's gitignored; leave it that way.
- Vercel functions are stateless — no real filesystem persistence. Use Upstash Redis (already wired) or email for anything that needs to persist.
- **Playwright is the site's test harness** (`@playwright/test`, official Microsoft package from `github.com/microsoft/playwright`). It is **local dev-only tooling**: no `/api/*` files are added, so the Hobby 12-function cap is untouched. Config is `playwright.config.js` (repo root); specs live in `tests/` (`smoke`, `auth`, `tools`, and the `meta.mirror` byte-identical guard). Browser binaries live outside git in `~/AppData/Local/ms-playwright` (re-fetch with `npx playwright install` on a fresh machine); `node_modules/` and Playwright report/result folders are gitignored. Specs assert gates, structure, and preparatory (no-login, no-API) actions — they must **never** trigger real AI completions (that spends Anthropic tokens). Tests navigate to `/index.html` explicitly because live `/` redirects to `home.html` while the local preview serves `index.html` at `/`.

## Workflow expectations
- **Testing is owned by Playwright, not by manual inspection. Your role is the fixer; Playwright is the tester.** After any code change, run `npm test` (the `tests/` suite across Chromium/Firefox/WebKit against the port-5500 preview, which proxies `/api/*` to live Vercel; the static server auto-starts). Playwright drives the site and reports failures — treat each failure as a bug ticket: reproduce → fix the source → re-run the failing spec → confirm green. Do not eyeball the preview by hand or `curl` endpoints as the primary check; let the suite tell you what's broken and fix what it reports.
- Only once `npm test` is green: `cp index.html "Latest Business card.html" && git add -A && git commit && git push` — Vercel deploys automatically. (Mirror is `index.html` → `Latest Business card.html` ONLY. Never `cp home.html index.html`. The `meta.mirror` spec fails the suite if the two drift, so a green run already guarantees the mirror is in sync.)
- After Vercel deploys, run `npm run test:live` (the same suite against `https://usman-s-first-website.vercel.app`) as a post-deploy smoke check; fix any regression it surfaces rather than assuming the deploy succeeded.
- **Grow the suite with the site**: when you build a new tool or fix a Playwright-reported bug, add or extend a spec in `tests/` so the regression can't silently return. Keep specs on resilient locators (`getByRole`/ids) and free of real AI-completion calls.
